## Errors resolved (in order)

1. **`styleGroupId` "Style SKU Count" (yesterday's open item).**
   Root cause: ids must **continue from the seller's existing catalog**, not start
   at 1. 8 products already listed ‚Üí new groups start at **9**. Fix: configurable
   `style_group_id_start` in `config/rules.yaml` (set to 9); `run.py` offsets
   `styleGroupId = start + i - 1`. The 2-SKU manual file with ids 9,10 passed
   structural validation and reached field-level (`SKU_VALIDATION_FAILED`) errors.

2. **Field-level errors** (from resubmission file `SYSTEM ERROR MESSAGE`):
   - `MRP cannot be empty or non numeric` and `ISP cannot be empty for DIY source`
     ‚Äî the manual file had them blank; our pipeline already fills MRP/ISP. Also
     learned Myntra wants these as **numeric cells**, not text.
   - `6 digit Pincode is missing in manufacturer/packer name and address` ‚Äî we wrote
     just "Ijor Ethnic Partners". Fix: full address with pincode in
     `config/constants.yaml` (‚Ä¶FARIDABAD‚Ä¶Haryana, 121006).

3. **`For the image column: ‚Ä¶, extension is not jpg/jpeg` (the last blocker).**
   Myntra checks the image URL extension **literally**; Shopify CDN URLs end in
   `.webp` (served as JPEG via content negotiation, but the string is `.webp`) ‚Üí
   rejected. Swapping the extension on Shopify 404s (no jpg variant stored).
   **Solution: host the converted JPGs on S3 and write `.jpg` URLs.**

## S3 image hosting (the real fix for images)

- Bucket `ijorethnicpartners` (ap-south-1). Uploaded converted JPGs to the
  `myntra/` prefix; made `myntra/*` public-read via a bucket policy
  (`S3/s3-bucket-policy-ijor-public-read.json`). Verified anonymous `200 image/jpeg`.
- IAM: scoped the uploading user with object put/get + list on the bucket. The
  committed policy was later trimmed to least-privilege
  (`S3/iam-policy-s3-image-upload.json`) ‚Äî see "Security review response" below.
- URL pattern: `https://ijorethnicpartners.s3.ap-south-1.amazonaws.com/myntra/<sku>/<n>.jpg`
  (per-SKU folder; S3 key mirrors the local `output/images/<sku>/<n>.jpg` layout)
  ‚Äî the key basename equals the local filename the pipeline already produces.
- Cost is negligible (~$0.001/mo storage; egress within the 100 GB/mo free tier).
  Myntra copies images to its own CDN during cataloging, so S3 is hit only briefly.

### Code audit + fixes
Ran a recall-biased review of the pipeline diff. Five findings; fixed the two that
bite in normal use:
- **#1 URL/upload decoupling:** a non-uploading run still emitted S3 URLs for images
  it never uploaded. Fix: one gate (`use_s3 = do_upload and public_base_url`) drives
  both; if not uploading, fall back to source URLs.
- **#2 whole-directory upload:** `upload_images` scanned all of `output/images/`,
  re-uploading stale images from earlier batches. Fix: it now takes explicit paths
  (this run's validated images only); `run.py` passes `img.passed`.
Findings #3 (duplicated prefix), #4 (`s3_bucket` KeyError), #5 (leading-zero HSN)
left as noted, low-risk.

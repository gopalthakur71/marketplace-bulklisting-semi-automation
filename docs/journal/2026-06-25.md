## Errors resolved (in order)

1. **`styleGroupId` "Style SKU Count" (yesterday's open item).**
   Root cause: ids must **continue from the seller's existing catalog**, not start
   at 1. 8 products already listed → new groups start at **9**. Fix: configurable
   `style_group_id_start` in `config/rules.yaml` (set to 9); `run.py` offsets
   `styleGroupId = start + i - 1`. The 2-SKU manual file with ids 9,10 passed
   structural validation and reached field-level (`SKU_VALIDATION_FAILED`) errors.

2. **Field-level errors** (from resubmission file `SYSTEM ERROR MESSAGE`):
   - `MRP cannot be empty or non numeric` and `ISP cannot be empty for DIY source`
     — the manual file had them blank; our pipeline already fills MRP/ISP. Also
     learned Myntra wants these as **numeric cells**, not text.
   - `6 digit Pincode is missing in manufacturer/packer name and address` — we wrote
     just "Ijor Ethnic Partners". Fix: full address with pincode in
     `config/constants.yaml` (…FARIDABAD…Haryana, 121006).

3. **`For the image column: …, extension is not jpg/jpeg` (the last blocker).**
   Myntra checks the image URL extension **literally**; Shopify CDN URLs end in
   `.webp` (served as JPEG via content negotiation, but the string is `.webp`) →
   rejected. Swapping the extension on Shopify 404s (no jpg variant stored).
   **Solution: host the converted JPGs on S3 and write `.jpg` URLs.**

## S3 image hosting (the real fix for images)

- Bucket `ijorethnicpartners` (ap-south-1). Uploaded converted JPGs to the
  `myntra/` prefix; made `myntra/*` public-read via a bucket policy
  (`S3/s3-bucket-policy-ijor-public-read.json`). Verified anonymous `200 image/jpeg`.
- IAM: scoped the uploading user with object put/get + list on the bucket. The
  committed policy was later trimmed to least-privilege
  (`S3/iam-policy-s3-image-upload.json`) — see "Security review response" below.
- URL pattern: `https://ijorethnicpartners.s3.ap-south-1.amazonaws.com/myntra/<sku>/<n>.jpg`
  (per-SKU folder; S3 key mirrors the local `output/images/<sku>/<n>.jpg` layout)
  — the key basename equals the local filename the pipeline already produces.
- Cost is negligible (~$0.001/mo storage; egress within the 100 GB/mo free tier).
  Myntra copies images to its own CDN during cataloging, so S3 is hit only briefly.

were built by minimal edits on the user's proven file rather than reformatting it.

## Evening — web-app design + Phase 1 backend execution

### Brainstormed and locked the listing web app + cloud/CI-CD design
Decided to wrap the working CLI pipeline in a beautiful-but-simple web UI so
non-technical teammates can use it, shipped via a full AWS cloud + CI/CD setup
(the showcase is the cloud story, not a developer/frontend profile). Key decisions:
- **Stack:** FastAPI + Jinja2 + Tailwind + htmx in ONE Docker container on a
  start/stop EC2 t3.micro (ap-south-1). Not Lambda (15-min/29-s limits, cold starts).
  Idle cost ≈ EBS only (~$0.5–0.75/mo); no Elastic IP.
- **Auth/secrets:** Cognito hosted-UI login + JWT validation; EC2 instance IAM role
  (no static keys); SSM Parameter Store (non-secret config) + Secrets Manager
  (Cognito client secret). No shared password.
- **CI/CD:** GitHub Actions on push to main → pytest gate → build image →
  authenticate to ECR via GitHub OIDC (no long-lived keys) → push. EC2 user-data +
  systemd pulls `:latest` on boot (start-on-demand = deploy newest).
- **Two flows:** (A) Generate Myntra sheet from Shopify CSV; (B) **Fix Myntra errors**
  interactively (option C) — non-tech user uploads a resubmission file, gets
  plain-language fixes (auto-fix / pick-from-dropdown / explain-only), regenerates.
- **styleGroupId ledger:** JSON-in-S3 (`state/myntra_groupid.json`), not a DB —
  `reserve()` records a pending batch without advancing the counter; `confirm()`
  ("Mark upload successful") advances it. Failed uploads free their IDs. DB only
  if this ever goes multi-tenant/SaaS.
Spec: `docs/superpowers/specs/2026-06-25-listing-web-app-cloud-deploy-design.md`.

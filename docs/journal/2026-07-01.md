## Error log — everything we hit, in order

### E1 — SSH private key "permissions too open" (WSL)
- **Symptom:** `ssh -i listing-app.pem …` → `Permissions 0777 for 'listing-app.pem' are too
  open … bad permissions … Permission denied (publickey)`.
- **Root cause:** the key lived on `/mnt/e/...`, a Windows NTFS drive mounted in WSL as DrvFs,
  which forces everything to `0777`; `chmod` **silently doesn't stick** there.
- **Fix:** opened the same key from **Git Bash** instead (worked immediately). Alternative
  documented: copy the key into the WSL ext4 home (`~/.ssh/`) and `chmod 600` there.

### E2 — User-data never ran (instance #1)
- **Symptom:** `systemctl status listing-app` → *"Unit … could not be found"*; `docker
  --version` → *command not found*. Nothing was installed.
- **Root cause (first instance):** user-data didn't execute. We created a clean, LF-only
  `aws/ec2/userdata.sh` to upload directly (avoid CRLF paste mangling).

### E3 — `ERR_CONNECTION_REFUSED` in the browser
- **Symptom:** browser to the public IP → *refused to connect*.
- **Diagnosis:** **refused ≠ timeout.** Refused means the host was reached but **nothing is
  listening on port 80** (as opposed to a SG block, which *times out*). Confirmed the container
  wasn't up.

### E4 — No outbound internet → `dnf` couldn't install Docker (instance #2)
- **Symptom:** relaunched; user-data *still* didn't produce Docker. `ping www.google.com` →
  100% packet loss.
- **Root cause:** the **security group's outbound (egress) rule** had been removed/altered.
  SGs are **stateful**, so inbound SSH still worked (the reply is allowed), but **new outbound
  connections** (`dnf`, `docker pull`) were dropped → `dnf install -y docker` failed → with
  `set -euo pipefail` the whole user-data script aborted. This was the real reason user-data
  "didn't run."
- **Fix:** re-added **Outbound: All traffic → `0.0.0.0/0`**. `ping` then succeeded.

### E5 — Container up but serving nothing (`curl localhost:80` refused)
- **Symptom:** after installing Docker manually and starting the service, `docker ps` showed
  the container "Up", **but** `curl -i http://localhost` → *Failed to connect to port 80*.
- **Root cause:** `docker ps` `COMMAND` column read **`python run.py`** — the **CLI batch
  entry, not the uvicorn web server.** Nothing listened on 8080 inside the container, so the
  `80->8080` mapping forwarded to a dead port. The running image was the **wrong (old) one.**

### E6 — ECR `:latest` was stale (the headline problem)
- **Symptom:** the box faithfully pulled `:latest`, but `:latest` had `CMD python run.py`,
  from before the FastAPI app existed. The repo `Dockerfile` correctly says
  `CMD ["uvicorn", "src.web.main:app", "--host", "0.0.0.0", "--port", "8080"]` (commit
  `8444761`), so the image in ECR was out of date.
- **Root cause:** **CI had failed on every run since the web-app merge.** GitHub Actions
  showed run #1 (`c03a6e5`, Jun 29) green — that built the old placeholder image now stuck as
  `:latest` — and **runs #2–#6 (Jun 30, the `ca42958` merge and after) all red.** Because
  `build-and-push` has `needs: test`, a red `test` job means the image is never rebuilt, so
  `:latest` froze at the last green build.

### E7 — The actual CI bug: `TypeError: unhashable type: 'dict'`
- **Symptom:** the failing job was **`test`** (deploy gate), on
  `tests/web/test_fix.py::test_fix_upload_shows_buckets`, raising `TypeError: unhashable type:
  'dict'` deep in Jinja2's template cache.
- **Root cause:** the **deprecated `TemplateResponse(name, context)` positional form** (used
  **8×** across `pages.py`, `generate.py`, `fix.py` — the exact nit logged back on the web-app
  build). It worked locally because the dev machine has **Starlette 0.38.6**, which only emits
  a `DeprecationWarning`. **CI installs deps unpinned**, so it pulled **Starlette 1.3.1**, where
  the old form is a hard error: it treats the template-name string as the `request` and the
  **context dict as the template name** → Jinja tries to use a dict as a cache key → crash.
  *This is why tests "passed locally" but CI was red — a silent local/CI dependency drift.*
- **Fix (`d566304`):** migrated all 8 calls to the current `TemplateResponse(request, name,
  context)` signature (and dropped the now-redundant `"request"` key from the context dicts).
- **Verification (did it properly):** local tests pass on any version, so they don't *prove*
  the fix. Built a **fresh venv with the latest unpinned deps** — resolved **Starlette 1.3.1 /
  FastAPI 0.138.2 / Jinja2 3.1.6**, i.e. CI's exact stack — and ran the full gate:
  **57 passed in 121s.** Fix confirmed against the environment that was actually failing.

### E8 — Browser still blank after the fix (transient)
- **Symptom:** once the correct image was running (`curl localhost` → 200), the **browser** at
  the public IP still timed out for a while.
- **Diagnosis / resolution:** purely the network path (app was healthy on localhost). It
  started working — on both browser and phone — after re-saving the SG inbound rules and a
  fresh browser load. Most likely a **cached failed connection** in Chrome plus the SG rule
  re-apply. Noted the isolation ladder for next time: `curl` from the laptop (bypass browser) →
  re-check SG port-80 source → confirm current public IP → test on phone/mobile data (rule out
  ISP blocking port 80).

### E9 — Console role/policy flow didn't match the runbook
- **Symptom:** Step 1's "create the inline policy while creating the role" didn't line up with
  the current console.
- **Fix:** had to **create the role**, **create the policy separately**, then **attach** it.
  Rewrote runbook Step 1 as explicit **A (create role) / B (create policy) / C (attach)**.

## Lessons / decisions

- **`refused` vs `timeout` is a fast diagnostic.** Connection *refused* = reached the host,
  nothing listening (app/port problem). *Timed out* = dropped in transit (SG/NACL/route/ISP).
- **Security groups are stateful** — inbound SSH working tells you nothing about outbound.
  A broken egress rule breaks `dnf`/`docker pull` while SSH still works.
- **`docker ps` COMMAND column is the truth** about which image is really running. `python
  run.py` vs `uvicorn …` instantly told us `:latest` was stale.
- **Unpinned `requirements.txt` caused a silent local/CI split** — this whole deploy detour
  traces back to it. **Follow-up (agreed, not yet done): pin `requirements.txt`** to the
  resolved versions and re-verify, so CI and local can't drift again.
- **Verify a fix against the environment that failed, not the one that passed.** Local Starlette
  0.38.6 would have "confirmed" a fix that might not work on CI's 1.3.1; the fresh-venv run is
  what actually proved it.

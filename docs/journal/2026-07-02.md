## The login bug chain (each one hid the next)

1. **`redirect_mismatch`.** The SSM param `cognito_redirect_uri` had a **trailing newline** —
   the app sent `…/auth/callback%0A` in the authorize URL, which Cognito rejected. Nasty to spot:
   bash `$(…)` strips trailing newlines, so a CLI read looked clean; only `--output json` revealed
   `"…callback\n"`. Fixed by rewriting the param; `settings.py` now `.strip()`s all values.

2. **`FORCE_CHANGE_PASSWORD`.** Login form rejected the password because the test user was still in
   the temporary-password state and the managed-login UI didn't surface the reset flow. Fixed with
   `admin-set-user-password … --permanent` → status `CONFIRMED`.

3. **`ERR_TOO_MANY_REDIRECTS` (the real one).** A tight server-side loop:
   `/auth/callback → 302 /login → Cognito → /auth/callback → …`, never reaching `/`. The callback
   was raising `AuthError` on every request and the handler bounces auth errors to `/login`. We
   isolated each callback stage by replaying a **freshly-captured unused auth code** (hit the
   authorize URL with `state=diag123` → app returns `{"detail":"invalid state"}` *before*
   exchanging, so the code stays valid) through `docker exec`. Token exchange: OK. JWT verify:
   **`JWTClaimsError: No access_token provided to compare against at_hash claim.`** Cognito
   id_tokens carry an `at_hash` claim, and python-jose validates it against the access_token unless
   told not to — but `verify_jwt` only has the id_token (the cookie holds no access_token). **Fix:**
   `jwt.decode(…, options={"verify_at_hash": False})` + a regression test that signs a real RS256
   id_token *with* `at_hash` (would have caught this originally). Committed `8fdc4e1`, CI rebuilt
   `:latest`, restarted the box → **login works**.

4. **`/generate` 500 → S3 `AccessDenied`.** After login, the generate page threw. The instance role
   had SSM + Secrets but **no S3** — reading the group-id ledger failed. Subtlety: without
   `s3:ListBucket`, a *missing* key returns 403 (not the 404 the code catches) → 500. Fixed by
   adding inline policy `listing-app-s3` (ListBucket + Get/PutObject on `ijorethnicpartners`);
   verified with the IAM policy simulator.

### Tooling gotcha worth remembering
Git Bash (MSYS2) rewrites any argument starting with `/` into a Windows path, so
`aws ssm get-parameter --name /marketplace-listing/…` silently became
`C:/Program Files/Git/marketplace-listing/…` → phantom `ParameterNotFound`. Prefix with
`MSYS_NO_PATHCONV=1`.

## CD: the pipeline now deploys

`ci-cd.yml` previously stopped at ECR (`test → build-and-push`); deploy was a manual
`systemctl restart` on the box. Added a third job **`deploy`**: assume the same OIDC role →
resolve the instance by `tag:Name=listing-app` → `ssm send-command "systemctl restart
listing-app"` → wait for `Success`. No inbound SSH, no static IP; a changed public IP doesn't
matter. Enabling AWS changes: attached `AmazonSSMManagedInstanceCore` to the instance role (it's
now SSM `Ping=Online`) and added an `ssm-deploy` inline policy to the OIDC role.
